# Модель угроз

Документ фиксирует базовую модель угроз для платформы `Игровая площадка ИВТ`, описывает ключевые активы, поверхности атаки, техники злоумышленника, текущие меры защиты и остаточные риски.

## 1. Границы системы

Система состоит из следующих контуров:

- `Frontend`
  React/Vite SPA, работает в браузере студента, преподавателя или администратора.
- `Backend`
  Go-сервис, который:
  - аутентифицирует пользователей;
  - выдаёт role-aware `catalog` и `runtime`;
  - принимает action-команды;
  - исполняет SQL в изолированной in-memory БД;
  - хранит состояние в `server-data/state.json`;
  - отправляет live-обновления по WebSocket.
- `Reverse proxy`
  `Caddy` в production-конфигурации завершает TLS и проксирует трафик.
- `Host security`
  `fail2ban` и внешние edge-механизмы защиты на уровне сервера или perimeter.

## 2. Защищаемые активы

Критичные активы:

- учётные данные преподавателей и администраторов;
- JWT-токены и сессии пользователей;
- скрытые решения задач и эталонные SQL-запросы;
- шаблоны БД, датасеты и SQL-схемы;
- результаты студентов, журнал действий, отправки и телеметрия;
- доступность сервиса на семинаре;
- целостность серверного `state.json`.

## 3. Роли и возможные нарушители

Внутренние роли:

- студент;
- преподаватель;
- администратор.

Потенциальные нарушители:

- студент, пытающийся получить скрытые решения;
- студент, пытающийся повлиять на чужие данные или результаты;
- внешний анонимный атакующий;
- злоумышленник с украденным JWT;
- злоумышленник с доступом к хосту;
- недобросовестный преподаватель или администратор.

## 4. Поверхности атаки

Основные точки входа:

- `POST /api/auth/login`
- `GET /api/bootstrap`
- `POST /api/reset`
- `POST /api/actions/{action}`
- `GET /ws`
- reverse proxy `80/443`
- импорт SQL-схем и датасетов
- запуск SQL в семинаре и playground
- локальное хранение токена в браузере

## 5. Основные сценарии угроз

### 5.1. Получение скрытых решений

Цель:
- студент пытается получить `expectedQuery`, `starterSql`, приватные задачи, лишние семинары или полный каталог пользователей.

Вектор:
- анализ frontend bundle;
- прямой вызов API;
- чтение websocket payload;
- эксплуатация ошибок сериализации данных.

Текущий статус:
- frontend больше не содержит клиентский `seed.ts`;
- сервер отгружает `catalog` по роли;
- студенту не отправляются `expectedQuery` и `starterSql`, если преподаватель не включил показ;
- студент видит только доступные ему live-семинары;
- `passwordHash` не отправляется в клиент.

Остаточный риск:
- преподаватель может вручную открыть референс;
- при наличии XSS на фронтенде скрытые данные и JWT остаются под риском.

### 5.2. Брутфорс входа

Цель:
- перебор логинов преподавателей и паролей администратора.

Вектор:
- массовые запросы к `POST /api/auth/login`.

Текущий статус:
- rate limit на логин по IP;
- одинаковые сообщения об ошибке входа;
- проверка пароля через constant-time compare;
- `Argon2id` для новых и мигрированных хэшей.

Остаточный риск:
- без внешнего WAF и fail2ban атака с распределённых IP остаётся частично возможной;
- студенческий режим по фамилии упрощён и не рассчитан на высокий уровень сопротивления enumeration.

### 5.3. Подмена origin / CSRF-подобные вызовы

Цель:
- выполнить запросы к backend из чужого origin.

Вектор:
- поддельные браузерные запросы с другого домена;
- попытка открыть WebSocket с нерелевантного origin.

Текущий статус:
- allowlist по `ALLOWED_ORIGINS`;
- проверка `Origin`;
- security headers и CSP.

Остаточный риск:
- при неверно настроенном `ALLOWED_ORIGINS` защита ослабляется;
- если токен украден вне браузера, origin-проверка уже не спасает.

### 5.4. Кража JWT и захват сессии

Цель:
- атакующий получает токен и использует его от имени пользователя.

Вектор:
- компрометация браузера;
- XSS;
- компрометация устройства;
- утечка логов или дампов памяти.

Текущий статус:
- JWT имеют срок жизни;
- backend проверяет алгоритм и подпись;
- роль и userId не доверяются клиенту без токена.

Остаточный риск:
- токен хранится в `localStorage`, а значит XSS остаётся критичной угрозой;
- нет rotate/revoke списка токенов;
- нет привязки токена к устройству.

### 5.5. SQL injection в SQL engine

Цель:
- обойти ограничения, выполнить опасные команды, получить доступ к системным возможностям SQLite или повесить сервис.

Вектор:
- злонамеренный SQL в семинаре или playground;
- импорт схемы с неожиданными SQL-конструкциями.

Текущий статус:
- SQL исполняется в отдельной in-memory SQLite БД;
- есть запрет опасных ключевых слов и служебных команд;
- есть лимит длины SQL и числа statements;
- есть ограничение числа строк результата;
- есть таймауты исполнения.

Остаточный риск:
- SQLite sandbox не эквивалентен отдельному контейнеру или отдельному PostgreSQL-инстансу;
- сложные запросы всё ещё могут создавать нагрузку на CPU;
- нет изоляции уровня ОС на каждую попытку.

### 5.6. Нарушение изоляции между студентами

Цель:
- один студент влияет на результаты другого.

Вектор:
- попытка использовать общую БД, общий runtime или чужие identifiers.

Текущий статус:
- каждый SQL-запуск идёт в отдельной in-memory БД на основании шаблона;
- draft сохраняется только в собственной области;
- backend валидирует доступ к семинару и задаче;
- role-aware выдача каталога ограничивает чужие сущности.

Остаточный риск:
- глобальный `state.json` остаётся общей точкой хранения состояния;
- при логической ошибке в backend возможны утечки через runtime/catalog.

### 5.7. Tampering с action endpoint

Цель:
- вызвать административные действия не своей ролью.

Вектор:
- ручной `POST /api/actions/{action}` с чужими параметрами.

Текущий статус:
- на backend есть `requireTeacher` и `requireAdmin`;
- валидируется принадлежность семинара/задачи;
- `DisallowUnknownFields` для auth-body;
- размеры action-body ограничены.

Остаточный риск:
- универсальный action endpoint сам по себе остаётся большим доверенным API-узлом;
- ошибка в одной из веток `performAction` может дать эскалацию прав.

### 5.8. DoS и деградация во время пары

Цель:
- положить backend или сделать его непригодным для семинара.

Вектор:
- спам actions;
- спам websocket-подключений;
- тяжёлые SQL-запросы;
- большой импорт схем.

Текущий статус:
- rate limits на login, action и websocket;
- ограничения размеров JSON и SQL;
- таймауты SQL;
- лимиты на datasets, hints, body и импортируемые SQL-скрипты.

Остаточный риск:
- при одном процессе и файловом state отказоустойчивость ограничена;
- нет очередей, worker isolation и autoscaling.

### 5.9. Компрометация `state.json`

Цель:
- изменить оценки, результаты, семинары, пользователей или логи.

Вектор:
- доступ к файловой системе сервера;
- подмена volume в Docker;
- компрометация контейнера.

Текущий статус:
- состояние хранится только на backend;
- в production рекомендуется отдельный volume.

Остаточный риск:
- нет криптографической защиты `state.json`;
- нет WAL/transaction layer уровня БД;
- нет разделения operational data и audit data.

## 6. Техники злоумышленника

Ниже перечислены практические техники, релевантные для этой платформы.

### T1. Enumeration учётных данных

Описание:
- перебор логинов преподавателей;
- перебор фамилий студентов;
- сравнение сообщений об ошибках.

Цель:
- подготовка к дальнейшему входу или социальной инженерии.

Защита:
- унифицированная ошибка входа;
- rate limit;
- скрытие списка фамилий из UI.

### T2. Password guessing / brute force

Описание:
- массовый перебор паролей администратора и преподавателей.

Защита:
- `Argon2id`;
- rate limit;
- fail2ban на reverse proxy логах;
- TLS в production.

### T3. Token replay

Описание:
- использование украденного JWT без знания пароля.

Защита:
- подпись JWT;
- срок жизни токена;
- origin control для браузерного использования.

Недостаток:
- нет ревокации;
- `localStorage` остаётся уязвимым для XSS.

### T4. Sensitive data exposure через client bundle

Описание:
- анализ JS bundle и API ответов ради скрытых решений.

Защита:
- удаление клиентского `seed.ts` из рабочего frontend flow;
- role-aware `catalog`;
- скрытие `expectedQuery`/`starterSql` для студента.

### T5. Abuse of generic action endpoint

Описание:
- отправка неожиданных action-полей или ручной вызов админских действий.

Защита:
- role checks;
- валидация payload;
- object ownership checks.

### T6. Resource exhaustion through SQL

Описание:
- длинные, рекурсивные или тяжёлые запросы;
- попытки вызвать дорогие операции на датасетах.

Защита:
- таймауты;
- statement limit;
- запрещённые SQL-команды;
- отдельный in-memory runtime на запуск.

### T7. Schema import abuse

Описание:
- загрузка слишком большого или вредоносного SQL-скрипта.

Защита:
- лимиты длины;
- лимиты числа датасетов;
- серверный импорт и валидация.

### T8. Horizontal privilege escalation

Описание:
- студент пытается открыть чужой семинар, задачу или черновик.

Защита:
- backend проверяет принадлежность семинара, задачи и пространства черновика;
- выбор семинара и задачи ограничен на сервере.

## 7. Текущие меры защиты

Приложение уже использует:

- `Argon2id` для новых и мигрируемых паролей;
- legacy-проверку `SHA-256` только для совместимости;
- JWT с проверкой алгоритма и подписи;
- allowlist по `ALLOWED_ORIGINS`;
- security headers и CSP;
- rate limiting:
  - login;
  - action endpoint;
  - websocket;
- строгую JSON-десериализацию;
- валидацию длины, формата и состава входных полей;
- role-aware API responses;
- скрытие чувствительных полей для студента;
- SQL ограничения и server-side sandbox;
- TLS через reverse proxy;
- fail2ban-шаблоны для host-level блокировки.

## 8. Остаточные риски

На текущем этапе остаются важные риски:

- токен в `localStorage`;
- SQLite вместо отдельного PostgreSQL sandbox;
- файловое состояние вместо production БД;
- отсутствие revoke/rotation для JWT;
- отсутствие полноценного WAF;
- отсутствие отдельной очереди/worker pool для тяжёлых SQL-задач;
- отсутствие криптографической защиты audit/state файла;
- упрощённый вход студентов по фамилии.

## 9. Приоритеты усиления

### Высокий приоритет

- перейти с `state.json` на PostgreSQL;
- вынести исполнение SQL в отдельный sandbox-контур;
- убрать JWT из `localStorage` в сторону `HttpOnly` cookie или другого более безопасного механизма;
- добавить ревокацию и принудительное завершение сессий;
- ограничить student login дополнительным кодом семинара или временным invite-механизмом.

### Средний приоритет

- добавить audit trail с неизменяемым хранилищем;
- выделить action endpoint в более узкие typed endpoints;
- добавить metrics/alerting по rate limit, auth failure, SQL timeout;
- внедрить внешнюю защиту на edge-уровне.

### Низкий приоритет

- персонифицированные device/session labels;
- экспорт security events в SIEM;
- тонкая аномалийная аналитика по подозрительным попыткам.
